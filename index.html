1 认识AI

🧠 一、LLM —— 大语言模型

▾

LLM 是所有大模型的统称，不是某一个具体的模型。就像"汽车"是一个品类，下面有丰田、宝马、特斯拉。

品类公司具体产品

两个特点：

特点含义
✅ 能力强能写论文、能总结、能翻译
⚠️ 只动口不动手只能输出文字，不能直接操作文件
LLM 是所有大模型的统称。你平时用的豆包、Kimi、DeepSeek、Claude，都是 LLM。

💬 二、Prompt —— 提示词

▾

Prompt 就是"你跟 AI 说的那句话"。 先给身份和场景，AI 就能给出更精准的回答。

差的 Prompt好的 Prompt
"帮我写论文""我是社会工作本科生，要写一篇医务社会工作的课程论文，3000字，需有文献综述"
"翻译一下""这是一篇医学论文摘要，请翻译成中文，保持学术风格"

原因：LLM 见过几乎所有场景。你告诉它你是谁、要什么，它就能匹配最合适的回答模式。

好 Prompt 的 5W1H：

要素意思你的例子
Who你是谁社会工作本科生
What你要什么课程论文
Where用在什么场景医务社会工作课程
Why目的是什么完成作业
How有什么要求3000字、学术风格
Prompt 就是"你跟 AI 说的那句话"。把背景交代清楚，AI 的回答就好用。

🔢 三、Token —— 计费单位

▾

Token 就是大模型世界的"字数"。

内容约等于
1 个汉字≈ 1.5 ~ 2 个 token
1 个英文单词≈ 1 ~ 2 个 token

快速估算：汉字按 1 字 ≈ 2 token 来算，基本够用。

为什么叫 Token？

大模型不认识汉字，只认识数字。Token 就是把文字切成小段、每段编个号：

"我是社会工作专业的本科生"
       ↓ 分词
["我", "是", "社会", "工作", "专业", "的", "本科", "生"]
       ↓ 转编号
[1024, 356, 7789, 4512, 8832, 128, 5631, 445]
       ↓ 模型只处理这些数字

一次对话大概花多少 Token？

场景大约 Token 数
一次简单问答几百 ~ 几千
读一篇论文几千 ~ 几万
写一篇课程论文几千
Token 就是大模型的"字数"，汉字大约 1 字 ≈ 2 token。厂商按 Token 收费，就像作文按字数算钱。

📝 四、Markdown 格式的优势

▾

同样一句话，三种格式的 Token 消耗完全不同：

以"医务社会工作概论"这句话为例：

Markdown：**医务社会工作概论**最少 ✅
约 8 token
HTML：<strong>医务社会工作概论</strong>中等 ❌
约 14 token
Word (.docx)：XML 格式说明书包装最多 ❌
约 40+ token

Word 为什么这么费 Token？

Word 文件本质是一个压缩包，里面全是 XML 格式说明书：

你看到的是： 医务社会工作概论（加粗）

Word 实际存的是： "正文" + "宋体" + "三号" + "加粗"
                 + "行距1.5倍" + "段前0.5行"
                 + "首行缩进" + "居中对齐" + ……

对比总结

对比项MarkdownWord
内容纯文字 + 少量符号文字 + 大量格式说明书
AI 能否直接读✅ 直接打开就能用❌ 需要转换
不同软件打开✅ 永远一样❌ Word/WPS 可能不一样
Token 消耗最少最多（3~10倍）
Markdown 省 token 不是因为魔法，而是它没有"格式包装纸"——内容就是内容，干净利落。给 AI 看，Markdown 是最优解。
2 使用方式

🌐 五、网页端 Chat

▾

打开浏览器就能用的 AI，最简单的使用方式。

国内主流产品对比

产品厂商特点适合做什么
豆包字节跳动界面简洁，回复快日常问答、资料总结
DeepSeek深度求索推理能力强，中文好论文思路梳理
通义千问阿里巴巴功能全面文档分析、长文本处理
Kimi月之暗面超长上下文，可一次读整本书读长篇论文、整理大量资料

网页 Chat 的核心局限

能力网页 Chat
回答问题✅ 能
上传文件让它分析✅ 能（手动上传）
直接读取你电脑上的文件❌ 不能
直接修改你电脑上的文件❌ 不能
需要你手动复制粘贴✅ 需要
网页 Chat = 你动手，它动口。你得自己上传、自己复制、自己粘贴。

🖥️ 六、IDE —— 工作台

▾

以 VS Code 为例说明——左边文件列表、中间编辑区、下面终端。

对比网页 ChatIDE 中的 AI
能问答✅✅
能读你电脑的文件❌ 需手动上传✅ 直接读取
能修改你电脑的文件❌ 只能输出文字✅ 直接修改
需要你复制粘贴✅ 需要❌ 不需要
流程步骤5~8 步1 步

同样"帮我修改论文第三段"：

网页 Chat： ① 打开论文 → ② 复制第三段 → ③ 粘贴到网页 ④ AI 给出新版本 → ⑤ 复制 → ⑥ 回到文件粘贴 → 6步，其中4步是你做的 IDE 中的 AI： ① 说"帮我改一下第三段" → AI 直接打开文件 → 修改 → 保存 → 1步
不是 IDE 功能更丰富，而是 IDE 省掉了"上传→下载→复制→粘贴"这些中间步骤。网页 Chat 能做的事，IDE 基本也能做，但 IDE 不需要你当"搬运工"。

🔑 七、API —— 打开模型的钥匙

▾

API Key 就是给程序用的"密码"。

网页 ChatAPI
怎么证明你是谁登录账号（手机号+验证码）API Key（一串密钥字符串）
谁在用人在操作程序在调用

网页端 Chat 也在背后调用 API

你在网页端 Chat 上打字发送 → 网页在背后调用厂商的 API → 只是厂商帮你管好了 API Key → 你不需要自己看到它
API Key 就是给程序用的"密码"，省掉了打开浏览器、登录、打字的步骤，让软件直接和 AI 对话。

🔧 八、MCP —— 工具箱协议

▾

MCP 是让一个模型可以使用很多个工具的协议。

MCP = 给模型配工具的"标准接口" ↓ 一个模型（比如 Claude）→ 通过 MCP → 可以用无数种工具 ├── 读文件 ├── 写文件 ├── 执行命令 └── 搜索代码

为什么配置 Agent 扩展时没感觉需要配置 MCP？

因为 Claude Code 这类 Agent 扩展默认已经配好了基础的 MCP 工具。

Claude Code 这类 Agent 扩展默认自带的 MCP 工具： ├── 文件系统操作（读文件、写文件） ├── 搜索（搜索项目文件） ├── 终端（执行命令） └── …… 你安装好 Agent 扩展后说"帮我看看这个文件" → 调用的就是默认的文件系统 MCP → 你已经在用 MCP 了，只是它是默认配好的
MCP 就是让 AI 能"动手"的工具箱协议。Agent 扩展默认已经配好了基础工具，你知道它是什么就够了，不需要自己去配置。
3 进阶玩法

📖 九、RAG —— 检索增强生成

▾

RAG 就是让 AI 在回答之前先"翻书"，基于找到的证据来回答，而不是凭记忆瞎编。

RAG 的工作流程

你提问 ↓ ① 检索：去知识库里搜相关的内容 ↓ ② 增强：把搜到的内容塞进 prompt ↓ ③ 生成：LLM 基于资料 + 自己的知识来回答

RAG 和网页端 Chat 不是二选一

网页 Chat = 使用方式（在浏览器里用） RAG = 技术方法（先检索再回答） 现在的网页 Chat 大多已经内置了 RAG： 你上传一篇论文到 Kimi → Kimi 先检索你上传的 PDF（RAG） → 再基于检索结果回答你

真正的区别：检索范围不同

网页 Chat（带RAG）IDE Agent（带RAG）
有没有 RAG✅ 有✅ 有
能搜索网络信息吗✅ 能（联网搜索）❌ 默认不能
能搜索本地文件吗❌ 只能搜你上传的✅ 能搜你电脑全部文件
RAG 就是让 AI 从"凭印象瞎说"变成"翻书再回答"。网页 Chat 翻的是网络和你上传的书，Agent 翻的是你电脑里的书——搜索范围不同。

🎯 十、Fine-tuning（微调）

▾

RAG vs Fine-tuning

对比RAGFine-tuning
模型本身变了吗❌ 没变✅ 变了
效果来源每次回答时检索资料模型"记住"了你的数据
适合场景知识问答、查资料特定格式、特定风格、特定任务
成本低（按token计费）高（训练需要算力）
灵活性✅ 资料随时换❌ 训练完就固定了
RAG 是给 AI 查资料，Fine-tuning 是给 AI 上课。对普通用户来说，RAG 更常用；只有需要"专属定制"时才考虑微调。

4 工具速查

🤖 十二、Agent —— 自主决策的助手

▾
LLM = 大脑（会思考） API = 钥匙（打开模型能力） MCP = 工具箱（能动手） IDE = 工作台（干活的地方） ↓ 以上全部整合 = Agent（自主决策的助手）

Agent 和普通 AI 调用的本质区别

普通 AI 调用Agent
你告诉它一步步指挥只说目标
它怎么做你说一步它做一步自己规划步骤
会不会循环❌ 一次完事✅ 思考→行动→观察→再思考
例子"先读这个文件，再总结给我""帮我整理这个文件夹"——它自己决定先读什么、再做什么

Agent 的核心循环

你派任务 ↓ Agent 思考：要完成这个任务，我需要什么？ ↓ Agent 行动：调用工具（读文件、搜资料、执行命令） ↓ Agent 观察：工具返回了什么结果？ ↓ Agent 再思考：结果够了吗？还需要做什么？ ↓ ……循环直到任务完成

常见的 Agent 产品

Agent一句话
Claude CodeVS Code 里的编程助手（你正在用的）
Codex CLIOpenAI 出的终端版 Agent
GitHub Copilot（Agent 模式）最早做代码补全的
Cursor一打开就是 AI 优先的编辑器
Agent = 大脑（LLM）+ 工具箱（MCP）+ 工作台（IDE）+ 自主决策。它不再等你一步步指挥，而是你说"要什么"，它自己想办法"做到"。

⚡ 十四、Ollama + Cherry Studio（离线方案）

▾

Ollama = 一行命令在本地跑各种开源模型的"管家"

以前在本地跑模型：折腾半天 Ollama：ollama run qwen2.5 → 自动下载 → 自动跑 → 直接对话

Cherry Studio = 一个"万能 AI 聊天桌面客户端"

一个软件里可以切换各种模型：Claude、GPT、本地 Ollama 跑的模型……不用打开多个网页。

通常搭配使用：Ollama（后台引擎）+ Cherry Studio（前台界面）

离线 Agent 最大的优势是免费、隐私、离线；最大的不足是模型能力弱、上下文短、速度慢。日常随便问问够用，正儿八经写论文还是云端模型更强。
∑ 完整链路串联
